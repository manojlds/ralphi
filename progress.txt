## Codebase Patterns
- Tools are registered in `extensions/ralphi/src/tools.ts` via `registerTools()` — each tool uses TypeBox schemas for parameters
- Tool logic should be extracted into separate files (e.g. `ask-tool.ts`) and imported by `tools.ts` for testability
- Mock-based tests use factories from `extensions/ralphi/test/factories/pi.ts` — `createMockUi()` accepts arrays of pre-queued responses for `confirm`, `input`, and `select`
- Pi UI API: `select()` returns `string | undefined` (single-select only), `input()` returns `string | undefined`, `confirm()` returns `boolean` — no native multi-select
- Multi-select is implemented via iterative `select()` calls with a "Done" sentinel option
- All quality checks via `npm run check` (lint + typecheck + tests); pre-commit hook enforces this
- Headless/non-interactive fallback pattern: tools that need UI should accept optional pre-supplied params (`providedAnswers`) and validate them; fail with actionable error + JSON example when neither UI nor pre-supplied params are available
- Skill-contract testing pattern: for prompt/skill-only behavior changes, add unit tests that read the relevant `skills/**/SKILL.md` file and assert required protocol markers/format fields to prevent accidental prompt regressions
